# Making Models Smaller: Distillation & Quantization

The two orthogonal levers for shrinking an LLM so it's cheaper/faster to run. **Distillation reduces the *number* of parameters; quantization reduces the *bytes per parameter*.** Often stacked together (a small distilled model, further quantized to 4-bit, is the recipe behind "runs on a laptop" models). Different axis from RAG/fine-tuning (those inject knowledge; these compress).

## Distillation — fewer parameters

Train a small **student** model to imitate a big, accurate **teacher**, rather than training from scratch on raw data.

**The core insight — soft targets:** the last transformer step (unembedding → softmax) produces a *full probability distribution* over the vocab. A hard training label is one bit ("the next word is `cat`"). The teacher's distribution (`cat` 60%, `dog` 25%, `hamster` 10%, `rock` 0.001%) carries far more — it encodes that cat/dog are both plausible, rock is absurd. The student learns from the *whole distribution*, absorbing the teacher's nuance.

- **Analogy:** answer key (hard label) says "B is correct." A good teacher says "B is correct, C was a tempting trap, D is nonsense" — the *why/how-close* is what transfers understanding.

**The three flavors** (the word "distillation" is stretched to cover all):

| Flavor | Signal used | Who can do it |
|---|---|---|
| Classic / soft-label | Teacher's full probability distribution | Model owner (needs logits) |
| Data / response | Only teacher's final **text** outputs | Anyone (just prompt→answer pairs) |
| Task-specific | Final text outputs, one narrow task | Anyone |

- **Important nuance:** only the *classic* flavor uses the distribution — the thing that makes the original 2015 (Hinton) term meaningful. Data/task "distillation" use only hard text outputs → mechanically **identical to ordinary supervised fine-tuning**; the *only* difference is the labels were generated by a teacher model rather than humans. Purists call these "synthetic-data fine-tuning."
- **Why the thin channel still works:** the teacher's *choice of token* still embeds its competence (a frontier model's answers are higher quality than cheap human labels). You can partly recover the lost richness by sampling the teacher many times per prompt (≈ sketching its distribution via a histogram of draws) or capturing its full chain-of-thought.

**Backend relevance:**
- The "mini"/"haiku"/"flash"-tier models you reach for in latency-sensitive paths are usually distilled from larger siblings → disproportionately good for their size. This is the "GPT-4 quality at a fraction of cost" mechanism.
- **Caveat:** distillation copies the teacher's mistakes/biases too. Student can't exceed teacher on what it's distilling. Imitation, not independent learning.
- ToS note: provider terms often restrict using their outputs to train competing models — check before doing data distillation commercially.

## Quantization — fewer bytes per parameter

Every parameter is a number. By default stored as **FP32** (4 bytes, ~7 digits precision) — but that precision is mostly wasted (0.4837192 vs 0.484 give nearly identical results). Quantization exploits the slack.

**The core idea:** store each weight in fewer bits by mapping the continuous range onto a small set of discrete buckets, + one shared **scale factor** per chunk.

```
weights in [-1, +1], want 8-bit (256 buckets):
  slice range into 256 even buckets
  store each weight as the INDEX of nearest bucket (1 byte, not 4)
  to use: index × scale → approximate float
  0.4837 → bucket 189 → recovers ~0.484   (rounding error = the cost)
```

**The precision ladder:**

| Format | Bytes/weight | Note |
|---|---|---|
| FP32 | 4 | full baseline |
| FP16 / BF16 | 2 | half precision; common training default (BF16 trades fraction-bits for range) |
| INT8 | 1 | classic target — ~4× smaller, little quality loss |
| INT4 | 0.5 | 8× smaller — makes "big model on one GPU/laptop" possible; sweet spot for self-hosting |
| ≤3-bit | < 0.5 | aggressive; noticeable degradation |

- A filename like `Llama-3-8B-Q4_K_M` literally encodes the quantization: Q4 = 4-bit.

**Why a backend engineer cares — it's ops math:**
- A 70B model: 140 GB (FP16) → 35 GB (INT4). Same param count, but now it *fits* on one GPU. This is the whole game for self-hosting.
- LLM inference is **memory-bandwidth-bound** — most time is spent moving weights from VRAM to compute units, not computing. Halving bytes/weight ≈ doubles throughput.
- **DDIA echo:** same reason columnar compression speeds up analytics — bottleneck is bytes read from storage, not CPU; smaller data = faster even with extra decode work.

**Two flavors (timing relative to training):**
- **PTQ (Post-Training Quantization)** — quantize finished weights. Cheap, no retraining; ~95% of downloadable quantized models. Blind rounding → low bit-widths can hurt.
- **QAT (Quantization-Aware Training)** — simulate rounding error *during* training so the model learns robust weights. More expensive, better quality at aggressive bit-widths.

**Caveat — outliers:** a few large-magnitude weights matter disproportionately; naively bucketing them with everything else is where quality drops. Modern methods (GPTQ, AWQ) protect important weights and quantize the rest harder. Rule of thumb: INT8 nearly free, INT4 sweet spot, below that you trade real quality for memory.

## The two together

- Distillation: shrinks *how many* weights.
- Quantization: shrinks *bytes per* weight.
- Orthogonal → stack them. Small distilled model + INT4 = cheap, fast, fits anywhere.
