### **Core Foundations: Open Addressing & Probing**

Before jumping into specific strategies, it is crucial to understand the foundational paradigm shared by these techniques: **Open Addressing**.

Unlike Chaining (which creates auxiliary structures like linked lists or trees to store collided items), **Open Addressing keeps all elements within the hash table array itself**. If a collision occurs at a primary slot, the algorithm hunts for alternative available slots using a **Probing Function**.

Mathematically, a probing function is expressed as:

$$P(k, i)$$

Where $k$ is the key and $i$ represents the **attempt number** (ranging from $0, 1, 2, \dots, m-1$, where $m$ is the total array size).

- **Attempt 0 ($i=0$):** Yields the _primary slot_ generated by the key's hash function.
    
- **Subsequent Attempts ($i > 0$):** Deterministically generate a sequence of permutations of the array indices until an empty slot is encountered.

### **1. Linear Probing**

[Watch Video](https://www.youtube.com/watch?v=5QKAXG25hig)

#### **The Core Concept**

Linear Probing is the most intuitive approach to open addressing. When a key's primary slot is occupied, the algorithm simply scans the array sequentially **one slot at a time to the immediate right**. If it hits the end of the array, it circles back to index `0` and resumes scanning.

#### **Mathematical Formula**

$$P(k, i) = (h(k) + i) \pmod m$$

- $h(k)$: The primary hash function.
    
- $i$: The linear attempt offset ($0, 1, 2, 3 \dots$).
    

#### **Visual Representation of Operations**

Imagine insertion of four keys ($k_1, k_2, k_3, k_4$) that all hash to the primary index **`2`**:

Plaintext

```
Initial Array Structure (Empty):
[ 0 ] [ 1 ] [ 2 ] [ 3 ] [ 4 ] [ 5 ] [ 6 ]
             
1. Insert k1 (hashes to 2) -> Slot 2 is free.
[ 0 ] [ 1 ] [ k1] [ 3 ] [ 4 ] [ 5 ] [ 6 ]

2. Insert k3 (hashes to 2) -> Slot 2 filled. Attempt 1: Moves right to 3. Free.
[ 0 ] [ 1 ] [ k1] [ k3] [ 4 ] [ 5 ] [ 6 ]

3. Insert k2 (hashes to 2) -> Slot 2 & 3 filled. Attempt 2: Moves to 4. Free.
[ 0 ] [ 1 ] [ k1] [ k3] [ k2] [ 5 ] [ 6 ]

4. Insert k4 (hashes to 2) -> Slots 2, 3, 4 filled. Attempt 3: Moves to 5. Free.
[ 0 ] [ 1 ] [ k1] [ k3] [ k2] [ k4] [ 6 ]
```

#### **Core Operations Internals**

- **Insertion:** Finds the first empty slot linearly and writes the key-value pair.
    
- **Lookup:** Hashes the key, travels to the primary index, and scans linearly matching keys. **Crucial Rule:** If it encounters a naturally **empty slot** during scanning, lookup stops immediately, meaning the key does not exist in the table.
    
- **Deletion (Soft Delete):** You **cannot** do a hard delete (leaving a clean empty slot), because doing so would break lookups for other keys that collided and slid past that slot. Instead, a **soft delete marker** (tombstone) is placed so lookups know to continue scanning forward.
    

#### **Why It’s Extremely Fast: Locality of Reference**

Even though linear scanning sounds inefficient ($O(N)$ worst-case), it achieves **near constant time $O(1)$ performance in average scenarios**. This happens because it beautifully leverages hardware and operating system behaviors:

1. When your CPU accesses a specific slot in RAM (e.g., `A[2]`), the OS brings a contiguous block of memory (often a 1KB/4KB block or line) into the **CPU Cache**.
    
2. Neighboring segments (like `A[3]`, `A[4]`, `A[5]`) are pulled into the cache simultaneously.
    
3. Scanning adjacent blocks happens directly within ultra-fast CPU caches, bypassing the slower main memory (RAM) lookup bottlenecks entirely.
    

#### **The Catastrophic Flaw: Clustered Collisions**

Linear Probing suffers severely from **Primary Clustering**.

- **How it happens:** If a few keys collide, they spill into adjacent slots. Any new incoming key that naturally hashes to _any_ slot within that filled block gets swept up into the collision, compounding the block size.
    
- **Result:** Independent keys start fighting for the same slots, degrading the hash table lookup into a slow, linear scan.

### **2. Quadratic Probing**

[Watch Video](https://www.youtube.com/watch?v=F-8pWiJv8ik)

#### **The Core Concept**

To break up the dense clusters formed by linear steps, Quadratic Probing abandons sequential $+1$ steps. Instead, it takes **progressively larger, quadratic leaps** across the array to find alternative spots.

#### **Mathematical Formula**

$$P(k, i) = (h(k) + c_1 \cdot i + c_2 \cdot i^2) \pmod m$$

- $c_1, c_2$: Arbitrary constant factors.
    
- **Constraint:** $c_2 \neq 0$ (otherwise it degrades back into Linear Probing).
    

As the attempt $i$ grows, the offset calculation expands quadratically ($1^2=1$, $2^2=4$, $3^2=9$, $4^2=16 \dots$), dispersing collided elements widely across the table.

#### **Visual Representation of Leaps**

Plaintext

```
Collision at Slot 2:
[ 0 ] [ 1 ] [ X ] [ 3 ] [ 4 ] [ 5 ] [ 6 ] [ 7 ] ... [ 11 ] ... [ 18 ]
             |       |                   |               |
             |       +-- i=1 (+1)        |               |
             +---------------------------+-- i=2 (+4)     |
             +-------------------------------------------+-- i=3 (+9)
```

#### **Trade-Offs & Limitations**

- **The Benefit:** It drastically **minimizes clustered and cascading collisions** compared to Linear Probing.
    
- **The Cache Penalty:** Because it jumps further away with each attempt, it loses some of the excellent **locality of reference** benefits that Linear Probing enjoyed. While the first couple of small jumps might still land inside the same CPU cache line, deeper quadratic leaps will force expensive random memory lookups from main RAM.
    
- **Secondary Clustering:** While it fixes primary clustering, it isn't completely immune to clustering. If two keys share the exact same primary hash index, they will follow the _exact same_ quadratic jump path, generating a secondary form of tracking clusters.

### **3. Double Hashing**

[Watch Video](https://www.youtube.com/watch?v=wV4K6fo0T58)

#### **The Core Concept**

Double Hashing completely obliterates fixed offset patterns (whether linear or quadratic) by making the size of the jump **dependent on the key itself**. It utilizes a **second independent hash function** to calculate the stepping interval.

#### **Mathematical Formula**

$$P(k, i) = (h_1(k) + i \cdot h_2(k)) \pmod m$$

- $h_1(k)$: Primary hash function determining the **starting position**.
    
- $h_2(k)$: Secondary hash function acting as the **step offset multiplier**.
    

#### **How It Works Mechanically**

Even if Key A and Key B collide at the same starting index via $h_1(k)$, their $h_2(k)$ values will most likely be completely different.

- Key A might take leaps of $+3$ slots on failure.
    
- Key B might take leaps of $+7$ slots on failure.
    
    They immediately separate along different paths, ensuring a beautifully sparse and uniform distribution.
    

#### **Strict Criteria for Choosing $h_2(k)$**

Selecting the secondary hash function requires precision to avoid breaking the hash table:

1. **Must never return `0`:** If $h_2(k) = 0$, then $i \cdot 0 = 0$, pinning the algorithm to the primary slot indefinitely and locking it in an infinite loop during collisions.
    
2. **Must cycle through the entire table:** The offset sequence must be mathematically capable of visiting every single slot in the array so that no space is left unreachable when the table gets full. _(Common implementation trick: Ensure the table size $m$ is a prime number, or make table size a power of 2 and ensure $h_2(k)$ always yields an odd number)_.
    
3. **Must be computationally fast:** Heavy algorithms like MD5 or SHA are unsuitable. It needs to be a fast, simple arithmetic hash function to keep lookup paths tight and efficient.
    

#### **Why Double Hashing is Highly Preferred**

- It provides the **highest uniform spread** of collisions across the entire address space.
    
- It is the **least prone to clustering problems** of any open addressing strategy.
    
- It yields highly optimized, short probing paths, minimizing overall average lookup latency.
